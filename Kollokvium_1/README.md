# Kollokvium 1 Project

Коллоквиум по ОС

## Практическая часть

### Инструкции по сборке

1. Создайте папку build: 
```
    mkdir build
    cd build
```

2. Настройте CMake:
```
    cmake ..
```

3. Соберите проект:
```
    cmake --build .
```

### Запуск приложения

```
    .\Debug\Kollokvium_1.exe
```

### Тестирование

Запуск тестов:
```
    .\Debug\tests.exe
```

Генерация отчета:
```
    cmake --build . --target allure_report
```

### Просмотр отчетов

Откройте в браузере: `Reports\allure-report\index.html`

### Возможности

- Вычисление факториала
- Удаление дубликатов из массива
- Реализация рекурсивного связного списка

### 1) ООП - полное определение

Объектно-ориентированное программирование - это одна из парадигм прогрммирования, 
которая основана на четырех основных принципах и идее создания классов и их объектов

**Наследование**

Наследование - принцип, который подразумевает создание иерархий класса с дочерними и родительскими классами.
К примеру, можно создать родительский класс человек и от него унаследовать мужчину и женщину, 
а от кого-то из них, в свою очередь, можно унаследовать студента, учителя и т. д.

**Инкапсуляция**

Инкапсуляция - принцип, который подразумевает сокрытие внутренних частей какого-либо класса
и написание интерфейса для взаимодействия с ними и с объектом. Можно привести очень простой пример. 
В классах почти все поля делают приватными, а доступ к ним организуют через публичные геттеры и сеттеры

**Полиморфизм**

Полиморфизм - принцип, который подразумевает, взаимодействие с несколькими объектами через один и тот же интерфейс

**Абстракция**

Этот принцип стал совсем недавно считаться основным на ряду с предыдущими тремя.
Пример абстракции можно привести такой:

Языки ассемблера это абстракция над соответствующими им процессорами, 
а языки программирования вроде C это еще более высокий уровень абстракции

### Объектно-ориентированная декомпозиция

Это метод выделения из большой и сложной части проекта частей поменьше,
а из них в свою очередь выделения еще более меньших частей и распределение их по соответствующим частям проекта.
Таким образом уменьшается сложность проекта и его легче становится понять


### Дизайн ПО

Дизайн - это определение архитектуры ПО, выделение основных компонентов и их взаимодействие между собой.

Есть много принципов дизайна, но вот основные из них:

**SOLID**

S - Single-Responsibility(Каждая отдельная часть ПО отвечает за что-то одно)
O - Open-Closed(Классы можно расширять, но нельзя менять для этого уже имеющееся)
L - Liskov substitution(При подстановке дочернего класса на место родительского, он должен выполнять все те же функция также)
I - Interface segregation(Большие части приложения следует разделять на части поменьше)
D - Dependency Inversion(Абстракция не должна зависить от реализации?)

**DRY**

Don't repeat yourself - принцип обозначающий, что не стоит дублировать некоторые вещи по несколько раз.
Таким образом, следует, к примеру, выделять повторяющиеся части кода в отдельные методы


**YAGNI**

You aren't gonna need this - принцип, говорящий о том, что не надо делать то, что тебе в данный момент не нужно

Также есть паттерны проектирования, их три вида: порождающие, поведенческие и структурные

Все эти вещи позволяют улучшить дизайн ПО
